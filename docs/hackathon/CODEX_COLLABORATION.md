# Codex Collaboration

## How Codex Was Used

Codex served as the execution partner for repository preflight, primary-source
research, Spec Kit initialization, specification and architecture, security
implementation, tests, documentation, UI construction, local browser
verification, repository publication preparation, and the Desktop-first
subscription sprint.

The majority of core product work was performed in one primary Codex thread so
the required session record represents the actual project. Bounded subagents
were used for official-rule and Spec Kit research, current Codex surface
research, donor-source inspection, contract drafting, isolated review, and the
Codex plugin/hook integration implementation and verification in a temporary
configuration. Sprint 002 also used bounded tasks for subscription-child
security review, Desktop fixture contracts, the delayed-attack matrix, UI
presentation, and documentation convergence. The primary thread reviewed and
integrated that bounded work.
The verified default branch was published at
`https://github.com/MahdiHedhli/verity-cordon` only after the clean-clone,
security, browser, and Spec Kit release gates passed.

## Operator Product Decisions

The operator selected:

- The Verity Cordon identity, descriptor, tagline, and public positioning
- Developer Tools as the primary track
- The Codex-first controlled memory-plane product boundary
- Selective revocation and replay as the recovery narrative
- A polished local Memory Control Room as a core judged surface
- Offline and live GPT-5.6 demo paths
- Codex Desktop as the primary human-facing demo, with CLI as the deterministic
  harness and offline fallback
- Supported ChatGPT subscription access as an explicit optional semantic
  provider, without treating a subscription as an OpenAI API credential
- A clean-room Trojan Hippo-inspired delayed-trigger scenario rather than a
  benchmark reproduction
- The hard post-hackathon deferral register

## Operator Security Decisions

The operator required:

- No durable trust before adjudication
- Provenance before persistence
- Deterministic policy as final authority
- Signed append-only events and testable cryptographic claims
- No secret-bearing model calls, telemetry, fixtures, or screenshots
- Explicit failures with no unverified injection
- Documented Codex surfaces only
- Adversarial, cross-session, streaming, and tamper tests
- Honest limitations for compromised hosts, native Codex internals, and factual
  truth
- No credential-file inspection, broad environment forwarding, provider
  substitution, or raw child-output logging in subscription mode
- A visibly lower `agentic_sandboxed` isolation label for subscription-backed
  Codex execution, with any observed tool event invalidating the result
- A fixed-data, local, inert demo sink whose reviewed implementation does not
  read host data or transmit externally. Digest pinning supports that code
  identity claim; the ordinary Python process is not an OS sandbox

## Corrections and Rejected Directions

- Generic snapshot rollback was rejected in favor of event-specific revocation
  and deterministic replay.
- Generic sliding-window streaming was rejected as the main contribution in
  favor of transactional writes and complete final evaluation.
- Manual editing of Codex memory files was rejected as a primary integration
  mechanism after current docs described those files as generated state.
- Semantic-model authority was rejected; GPT-5.6 produces structured evidence
  and deterministic policy decides.
- Treating the Codex subscription path as equivalent to the direct Responses
  API was rejected. Subscription execution is separately labeled, more tightly
  bounded at the process boundary, and makes no tool-free claim.
- Driving Codex Desktop UI as a hidden semantic API was rejected. The supported
  one-shot Codex runtime uses the same ChatGPT sign-in while Desktop remains the
  visible task and demo surface.
- Running or copying the Trojan Hippo benchmark was rejected. The demo uses an
  original fixed synthetic fixture and makes no benchmark-performance claim.
- Broad framework and enterprise scaffolding was rejected to preserve the
  Codex vertical slice.
- Public SaaS deployment was rejected for the hackathon threat boundary; the
  Control Room is loopback-only by default.
- Donor comparison language was corrected after inspecting live source, tests,
  published packages, and broken current-main streaming behavior.

## Runtime Contribution Recorded in Sprint 002

On 2026-07-15, the primary thread exercised one sanitized synthetic semantic
assessment through Codex CLI `0.144.4` under supported ChatGPT sign-in, without
`OPENAI_API_KEY`. The first explicit `gpt-5.6` request failed content-safely as
unavailable for that identity and did not fall back. The explicitly selected
`gpt-5.6-luna` request then returned a schema-valid
`live_codex_subscription` assessment in 11,026 ms with isolation labeled
`agentic_sandboxed` and a `quarantine` recommendation. Deterministic policy
remained final authority, and no raw child output or credential material was
retained.

This is evidence for one host, sign-in, model, synthetic input, and date. It is
not evidence of universal entitlement, stable latency, tool absence inside the
Codex binary, a completed Desktop UI rehearsal, or protection against a
compromised host. The successful run preceded the exact incremental JSONL and
release-review hardening. A post-hardening probe was externally rate-limited
and returned retryable, content-safe `failed/process_exit` with clean cleanup
state; that proves fail-closed failure handling, not a successful hardened live
completion or remote-model attestation.
